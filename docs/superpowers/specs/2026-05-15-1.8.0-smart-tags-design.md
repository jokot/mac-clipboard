# MaClip 1.8.0 — Smart Auto-Tagging — Design

**Date:** 2026-05-15
**Status:** Approved (pending implementation plan)
**Target version:** 1.8.0 (build 11)
**Branch:** feature/1.8.0-smart-tags

## Summary

Build the second pillar of MaClip's "private + intelligent local clipboard" wedge: automatically detect what kind of content each clip holds (URL, email, phone, JSON, code, hex color, diff) and surface that as small SF Symbol badges in the row, with a `tag:` search prefix that mirrors `from:` for filtering by source app.

All detection is computed at render time from existing item content — no new persisted fields, no schema migration. Legacy clips auto-tag without backfill. Detection is local, pure, and synchronous.

## Wedge / Positioning

The 1.7.x line shipped privacy (exclusion, concealed handling, fresh-install re-seed) and provenance (source icons, OCR badge, `from:` filter). 1.8.0 is the first piece of *intelligence*: MaClip starts understanding what's in a clip. Future 1.x releases extend the pattern (tag-based actions, content-aware previews, eventually on-device LLM transforms on macOS 26+). All processing remains on-device.

## Scope

**In scope:**
- New `Utilities/ContentTags.swift` with a `ContentTag` enum and pure detection function over `ClipboardItem`.
- Seven tags: `url`, `email`, `phone`, `json`, `code`, `color`, `diff`.
- Tag badge rendering in `ClipboardItemRow` (left of the OCR badge slot, before source icon).
- Search filter syntax extension: `tag:json`, `tag:code`, etc. Combines with the existing `from:` and free-text tokens.
- Detection at render time (no `ClipboardItem` field, no persistence change).
- In-memory cache per `ClipboardItem.id` to avoid recomputing on every redraw.

**Out of scope:**
- Persisted tags on `ClipboardItem`.
- User-defined tags / manual tagging.
- Tag-based actions ("Open URL", "Copy JSON without whitespace", etc.) — defer to 1.8.x.
- On-device LLM tag inference — defer until macOS 26 / Foundation Models batch.
- Backfill / migration — N/A because tags aren't persisted.

## Data Model

No persistence changes. `ContentTag` is a render-time concept:

```swift
enum ContentTag: String, CaseIterable {
    case url, email, phone, json, code, color, diff

    var symbolName: String {
        switch self {
        case .url:   return "link"
        case .email: return "envelope"
        case .phone: return "phone"
        case .json:  return "curlybraces"
        case .code:  return "chevron.left.forwardslash.chevron.right"
        case .color: return "paintpalette"
        case .diff:  return "plus.slash.minus"
        }
    }

    var displayName: String {
        switch self {
        case .url:   return "URL"
        case .email: return "Email"
        case .phone: return "Phone"
        case .json:  return "JSON"
        case .code:  return "Code"
        case .color: return "Color"
        case .diff:  return "Diff"
        }
    }
}
```

## Detection Rules

`ContentTagDetector.tags(for: ClipboardItem) -> Set<ContentTag>`. Pure function over the item's content. Image items get no tags. URL items always tag as `url`. Text items tag with whichever rules match.

```swift
@MainActor
enum ContentTagDetector {
    private static var cache: [UUID: Set<ContentTag>] = [:]

    static func tags(for item: ClipboardItem) -> Set<ContentTag> {
        if let cached = cache[item.id] { return cached }
        let detected = computeTags(for: item)
        cache[item.id] = detected
        return detected
    }

    static func clearCache() {
        cache.removeAll()
    }

    private static func computeTags(for item: ClipboardItem) -> Set<ContentTag> {
        switch item.content {
        case .url:
            return [.url]
        case .image:
            return []
        case .text(let text):
            return detectTextTags(text)
        }
    }

    private static func detectTextTags(_ text: String) -> Set<ContentTag> {
        var tags: Set<ContentTag> = []
        if matchesURL(text)   { tags.insert(.url) }
        if matchesEmail(text) { tags.insert(.email) }
        if matchesPhone(text) { tags.insert(.phone) }
        if isJSON(text)       { tags.insert(.json) }
        if matchesColor(text) { tags.insert(.color) }
        if looksLikeDiff(text){ tags.insert(.diff) }
        // Code last; suppress when JSON applies so JSON-only items don't carry both.
        if !tags.contains(.json) && looksLikeCode(text) { tags.insert(.code) }
        return tags
    }
}
```

**Per-tag rules:**

- **url**: a single trimmed token matches `URLUtils.linkURL(from:)` (the existing helper). Whole-string match after trim.
- **email**: matches `^[\\w.+-]+@[\\w-]+\\.[\\w.-]+$` after trim. Single-token only.
- **phone**: matches `^[+()0-9\\s\\-]{7,20}$` after trim AND contains at least 7 digits.
- **json**: text trimmed starts with `{` or `[` AND `JSONSerialization.jsonObject(with:)` succeeds. Catches both objects and arrays.
- **color**: matches `^#([0-9A-Fa-f]{3}|[0-9A-Fa-f]{6}|[0-9A-Fa-f]{8})$` after trim.
- **diff**: text contains at least 2 lines and starts with `diff --git`, OR has at least 2 lines starting with `+` or `-` followed by space at line start.
- **code**: heuristic. Score 0 baseline. +1 for each of: text has 3+ lines; contains balanced `{}`/`()`; contains common keywords (`function`, `def`, `class`, `import`, `let `, `var `, `const `, `return `, `for `, `while `, `if `, `else`, `=>`); has indentation (2+ leading spaces or tab on at least 2 lines). Threshold: score ≥ 2 → tag as code. Suppressed if JSON also matched.

All regexes use `try? NSRegularExpression(...)` compiled once as `static let`. Compilation failure means "no match" for that tag.

## Cache

The cache is keyed on `ClipboardItem.id` (UUID). Once an item is detected, it stays cached for the app session. Items removed from history are not actively evicted (memory leak is negligible — a Set of up to 7 raw-string-backed enums × per-item UUID ≈ kilobytes even with thousands of clips).

`MainActor` guards prevent concurrent writes to the cache. Reads from filtering and row rendering both happen on the main actor.

`clearCache()` is exposed for `ClipboardListViewModel.clearHistory()` to call as a best-effort cleanup.

## Row UI

`ClipboardItemRow`'s badge area currently renders (depending on flags / content type): `[ ConcealedBadge ] [ RevealButton ] [ OCRBadge ] [ SourceIconBadge ]` in the trailing area. Insert `TagBadgesView` immediately before `OCRBadge`:

```
[ content preview ............ ] [ tags ] [ OCR? ] [ source ]
```

`TagBadgesView`:

```swift
private struct TagBadgesView: View {
    let tags: Set<ContentTag>

    var body: some View {
        if !tags.isEmpty {
            HStack(spacing: 4) {
                ForEach(Array(tags).sorted(by: { $0.rawValue < $1.rawValue }), id: \.self) { tag in
                    Image(systemName: tag.symbolName)
                        .font(.system(size: 12, weight: .medium))
                        .foregroundColor(.secondary)
                        .frame(width: 16, height: 16)
                        .contentShape(Rectangle())
                        .help(tag.displayName)
                }
            }
        }
    }
}
```

Each subview (`TextItemContent`, `URLItemContent`, `ImageItemContent`) computes the tag set via `ContentTagDetector.tags(for: item)` and renders `TagBadgesView(tags: tags)` in the trailing area.

Sorted alphabetically by raw value to keep badge order stable across renders.

**URL items** always show the `url` badge — slight redundancy with the source-app icon but a useful visual cue.

## Search Filter Extension

Extend `ClipboardListViewModel.filteredItems` to parse `tag:` tokens. Existing grammar:
- `from:<value>` → source app filter (OR'd across multiple `from:` tokens)
- `<value>` → text filter (AND'd)

Adding:
- `tag:<value>` → content tag filter (AND'd across multiple `tag:` tokens; ALL must apply)

Example queries:
- `tag:code` → all items detected as code
- `tag:json hello` → JSON items containing the substring "hello"
- `tag:url from:Safari` → URL items copied from Safari
- `tag:email tag:json` → items detected as both email AND JSON (edge case; rare)

Parsing:

```swift
for token in trimmed.split(separator: " ", omittingEmptySubsequences: true) {
    let s = String(token)
    let lower = s.lowercased()
    if lower.hasPrefix("from:") {
        let value = String(s.dropFirst(5))
        if !value.isEmpty { sourceFilters.append(value) }
    } else if lower.hasPrefix("tag:") {
        let value = String(s.dropFirst(4)).lowercased()
        if let tag = ContentTag(rawValue: value) {
            tagFilters.append(tag)
        }
        // unknown tag values are ignored (no items will match if user types `tag:nonsense`)
    } else {
        textFilters.append(lower)
    }
}
```

Filter loop adds:

```swift
if !tagFilters.isEmpty {
    let tags = ContentTagDetector.tags(for: item)
    for required in tagFilters where !tags.contains(required) {
        return false
    }
}
```

Tag filters AND'd among themselves; compose naturally with `from:` (OR among source filters) and free-text (AND).

Search-field placeholder updated:

```
Search… (try "from:Safari", "tag:code", "tag:json hello")
```

## Migration

No persistence change. No migration. Legacy clips render with auto-detected tags immediately.

## Edge Cases

1. **Empty text** → empty tag set. `TagBadgesView` renders nothing. Row layout unchanged.
2. **Whitespace-only text** → trim is empty → no tags.
3. **JSON-looking text that fails to parse** (`{not valid json`) → no JSON tag. Avoids false positives.
4. **Very long text (100k+ chars)** — JSON parse + regex scans could be slow. Mitigation: truncate text to first 64 KB before tag detection; code/diff regex run on first 16 KB only. Documented inline.
5. **Color hex inside a paragraph** → only whole-token match (after trim); a paragraph mentioning `#FF0000` does NOT tag as color. Same for email and phone — single-token only.
6. **URL items always include `url` tag** → expected. Source icon and URL badge convey different info.
7. **OCR result clip** (`isOCRResult == true`) → tagged based on extracted text content; OCR badge and tag badges both render. Both are informative.
8. **Concealed item** → text content is masked from preview but real content is still in memory available to the detector. Tag badges still appear; preview stays masked. No privacy regression (content already exists in memory either way).
9. **Cache collision impossible** — UUIDs are unique per item.
